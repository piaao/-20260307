# GitHub 提交频率与巡检规范（项目经理）

- 文档版本：v1.1
- 生效时间：2026-03-07
- 适用项目：古代县令模拟器（`/home/admin/.openclaw/workspace`）
- 责任角色：项目经理·纪衡

## 1. 目标

1. 保证研发过程持续可追溯（避免长时间无提交）
2. 支撑站会/日报的版本核对（commit hash 可回溯）
3. 发现进度异常并提前预警（而非临近节点才暴露）

## 2. 提交频率硬规则

1. 工作时段（09:00-23:00）内，**每 2 小时至少 1 次提交**。
2. 每日站会后 30 分钟内，必须完成当日首个有效提交。
3. 每日日报前（18:00）必须至少有 1 次当日提交。
4. 若当日无有效变更，必须新增说明文件：`work/pm/no_change_reason.md`。

## 3. 风险分级

- 正常：距上次提交 ≤ 120 分钟
- 警告：距上次提交 > 120 分钟
- 高风险：距上次提交 > 240 分钟
- 里程碑偏差：连续 2 天未满足“日报前提交”要求

## 4. 提交粒度规范

1. 单次提交建议 3-20 个文件。
2. 每次提交只聚焦一个目标（策划/开发/测试/音效/流程之一）。
3. 禁止将“目录重构 + 功能改动 + 大量文档改写”混成单次提交。

## 5. Commit Message 规范

- `feat(role): ...` 新功能/新产出
- `fix(role): ...` 缺陷修复
- `docs(role): ...` 文档变更
- `chore(pm): ...` 流程或配置调整

示例：
- `feat(planner): add county-case progression table`
- `fix(developer): correct main scene path`
- `docs(pm): update milestone gating rules`

## 6. 自动巡检输出

项目经理自动任务每次运行时，必须输出以下字段：

- 最近 24h 提交次数
- 最近一次提交时间
- 距上次提交分钟数
- 2h/4h 规则是否违规
- 当前风险状态（正常/警告/高风险）

输出文件：
- `work/pm/commit_frequency_check.json`
- `automation/reports/daily_report.md`（“提交频率检查”栏目）

## 7. 执行说明

- 本规则为项目节奏基础约束。
- 若用户临时要求高频推进，可临时提升到“每 1 小时至少 1 提交”。
- 如需变更规则，必须先更新本文件，再在日报中注明“规则变更生效时间”。
