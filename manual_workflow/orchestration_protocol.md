# 多角色编排协议

## 1. 角色包定义

每个角色执行前，都必须明确以下六项：

- 本轮目标
- 允许读取的输入
- 允许写入的输出
- 禁止覆盖的文件
- 解锁条件
- 预计处理时长

## 2. 当前角色编排包

### PM
- 目标：刷新站会、依赖、提交频率、风险
- 输入：全局项目状态、Git、闸门、评分
- 输出：`work/pm/`
- 禁止覆盖：业务产物正文
- 解锁条件：始终可运行
- 时长：5-10 分钟

### Planner
- 目标：维护策划三件套、文案集、规则一致性
- 输入：站会、PM 风险、玩家评测
- 输出：`work/planner/`
- 禁止覆盖：玩家评分文件、开发代码
- 解锁条件：PM 已刷新当日基线
- 时长：15-30 分钟

### Player
- 目标：做真实玩家/市场视角评测
- 输入：策划三件套、文案集、里程碑目标
- 输出：`work/player/`
- 禁止覆盖：策划源文件
- 解锁条件：策划闸门通过
- 时长：10-20 分钟

### Designer
- 目标：主视觉、切图、UI联调约束
- 输入：策划定稿、玩家建议
- 输出：`work/designer/`
- 禁止覆盖：策划正文、评分文件
- 解锁条件：玩家评分 >= 9.0 或用户强制推进
- 时长：10-20 分钟

### Developer
- 目标：核心循环、数据接入、可测包
- 输入：策划定稿、设计约束、音效接入点
- 输出：`work/developer/` + `game/`
- 禁止覆盖：评分文件、PM里程碑文件
- 解锁条件：玩家评分 >= 9.0 或用户强制推进
- 时长：15-30 分钟

### QA
- 目标：静态检查 / 运行态冒烟 / 缺陷分级
- 输入：开发状态、版本说明、测试基线
- 输出：`work/qa/`
- 禁止覆盖：代码、策划正文
- 解锁条件：玩家评分 >= 9.0 且存在可测 demo
- 时长：10-20 分钟

### Sound
- 目标：音效方向、SFX清单、接入点
- 输入：策划场景、设计风格、开发触发点
- 输出：`work/sound/`
- 禁止覆盖：代码与评分
- 解锁条件：玩家评分 >= 9.0 或用户强制推进
- 时长：10-15 分钟

## 3. 串行调度算法

每轮只做一件事：

- 若站会过期 -> 先跑 `pm`
- 若策划未过闸门 -> 跑 `planner`
- 若策划已改动且评分未更新 -> 跑 `player`
- 若评分未达标 -> 继续 `planner`
- 若评分 >= 9.0 且需要视觉约束 -> `designer`
- 若评分 >= 9.0 且要落地功能 -> `developer`
- 若评分 >= 9.0 且 demo 可测 -> `qa`
- 若交互点稳定 -> `sound`

## 4. 冻结策略

默认冻结：`designer/developer/qa/sound`

解除冻结条件：
- 玩家评分 >= 9.0
- 或用户明确要求推进某角色

## 默认修改策略

- 默认增量修改：所有角色在原项目或原文档基础上修改。
- 非特殊说明不重写：除非用户明确要求，否则不做整份推倒重写。
